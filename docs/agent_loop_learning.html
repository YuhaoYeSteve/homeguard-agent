Security Agent · Agent Loop Deep Dive

从一次聊天请求看懂 Agent Loop

这个页面用真实代码解释本项目的 Agent Loop：模型只负责做 JSON 决策，后端负责校验、执行工具、把工具结果写回上下文，再让模型基于观察结果继续决策。 你可以把它理解成一个安防调度台：模型是调度员，工具是摄像头、搜索系统和联网检索，后端循环负责把每一步记录清楚。

这是核心循环。注意这里的“步数”是模型决策步：每一步模型输出要么是最终回答，要么是一次工具调用。

模型输出约束

模型必须只输出 JSON。后端用 raw_step.get("type") 区分 final_answer 和 tool_call。

观察回填机制

工具执行后，Loop 会追加两条消息：一条 assistant 记录刚才的 tool_call，一条 tool 记录工具结果。下一次模型调用能看到这些观察。

6. 代码走读

建议按下面顺序读，不要一开始就跳进工具实现细节。先抓住“循环骨架”，再看工具。

入口 chat.py:13-36

API 层不是 Agent，它只负责把 HTTP 请求变成一次 loop.run() 调用，并在响应后保存会话历史。

    model_client=ArkCLIModelClient(),
    tool_registry=ToolRegistry(),
)
response = loop.run(conversation_id, history, request.message)

这里是最重要的代码：初始化状态，最多循环 3 次，每次让模型输出 JSON，再按 type 分支。

    raw_step = self.model_client.generate_json(messages, schema_name="agent_step")
    if raw_step.get("type") == "final_answer":

    if raw_step.get("type") == "tool_call":

工具调用后，Loop 把“模型刚刚想调用什么”和“工具实际返回什么”都写回消息列表，这是下一轮模型能基于工具结果回答的关键。

7. 三条运行路径

下面用 Tab 对比三类请求。先看“模型直接回答”，再看“IoT 工具”，最后看“视频搜索工具”。

路径 A：用户说“你好”

模型第一次输出 final_answer，循环直接返回，不调用任何工具。

{
  "type": "final_answer",
  "answer": "你好，我可以帮你查看监控或控制摄像头。",
  "iot_action": "none"
}

测试证据：test_agent_loop.py:100-129 验证 registry.calls == []。

路径 B：用户说“把摄像头转向门口”

模型第一步输出 tool_call，ToolRegistry 调用 iot_control，Loop 把工具观察写回 messages，模型第二步再输出最终回答。

// 第 1 步：模型决策调用工具
{
  "type": "tool_call",
  "tool_name": "iot_control",
  "arguments": {
    "device_id": "camera_living_room",
    "action": "move",
    "target": "front_door"
  },
  "reason": "用户要求摄像头转向门口"
}

// 第 2 步：模型看到工具结果后最终回答
{
  "type": "final_answer",
  "answer": "摄像头已转向门口。",
  "iot_action": "move"
}

测试证据：test_agent_loop.py:50-98 验证模型被调用 2 次，第二次 messages 内包含 assistant 和 tool。

路径 C：用户说“找一下有人经过门口的视频”

模型应输出 video_search 工具调用，VikingDB 工具返回 VideoSearchResult[]，Loop 把它放到 video_results 顶层，前端可以直接展示视频搜索结果。

{
  "type": "tool_call",
  "tool_name": "video_search",
  "arguments": {
    "query": "有人经过门口",
    "limit": 10
  },
  "reason": "用户要查监控视频内容"
}

相关实现： 搜索常量、 请求体与签名请求、 结果解析。

8. 调试观察点

Agent Loop 调试的关键是“每一步输入输出都要能解释”。下面这些点最适合加日志或下断点。

你想确认什么 看哪里 正常现象 异常时怎么想
API 有没有把历史传进 Loop chat.py:15-25 history = session_store.list_messages(conversation_id) 多轮对话失效时，先看 conversation_id 是否复用。
模型收到哪些 messages loop.py:119-130 system prompt、tool spec、history、当前 user 依次出现。 如果模型乱调用工具，先看 TOOL_SPEC 是否描述清楚。
模型输出 JSON 是否可解析 ark_cli_client.py:108-126 顶层必须是 dict，且后续 type 是 final_answer 或 tool_call。 如果返回 Markdown 或自然语言，会变成 MODEL_JSON_PARSE_FAILED。
工具有没有被调用 tool_registry.py:18-46 tool.run(arguments, step) 返回 result 和 ToolEvent。 unknown tool 会得到 failed event；要检查 prompt 和 ToolName 枚举是否同步。
工具结果有没有进入下一轮模型 loop.py:132-157 messages 新增 assistant tool_call 和 tool observation。 模型最终回答没引用工具结果时，检查 observation 是否包含足够字段。
前端能不能拿到结构化结果 schemas.py:70-76 iot_state、video_results、tool_events 出现在 ChatResponse。 新工具若要前端专门展示，通常需要新增 response 字段和前端类型。
常见错误：为什么工具执行失败后 Loop 没有立刻报错？

当前设计是：ToolRegistry 捕获工具异常，返回 status="failed" 的 ToolEvent。 AgentLoop 仍会把失败观察写回 messages，让模型下一步可以给用户一个可读解释。 这比直接抛 500 更适合 Agent 产品，但也要求工具输出里的错误信息足够清楚。

常见错误：为什么最多 3 步后会返回兜底文案？

AgentLoop.max_steps = 3 是为了避免模型反复调用工具不结束。 如果 3 次模型决策都没有输出 final_answer，代码会返回“还没有得到稳定的最终回答”的兜底文案。 位置：loop.py:90-99。

10. 如何定制修改

真要改 Agent Loop，不建议一上来重写循环。先判断你的需求属于哪一层。

只改模型行为

改 prompts

例如让模型更积极使用视频搜索，优先看 backend/app/agent/prompts.py。 这种修改风险较低，但要用测试约束输出 JSON。

新增工具

改 schema + tool + registry + prompt

新增工具时，要同步 ToolName、TOOL_SPEC、工具类、ToolRegistry.tools，并补一个 Agent Loop 测试。

改循环策略

谨慎改 AgentLoop

例如允许多工具并行、调整最大步数、失败后提前终止，都属于循环策略修改。改前先写清楚状态变化和测试用例。

替换模型入口

保持 ModelClient 接口

业务代码只依赖 generate_json(messages, schema_name)。如果替换模型，尽量保持这个接口，避免 AgentLoop 感知具体模型 SDK。

新增工具的最小清单

步骤 文件 你要做什么
1 schemas.py 把新工具名加入 ToolName，必要时新增结果 schema。
2 prompts.py 告诉模型什么时候调用该工具，以及 arguments JSON 格式。
3 backend/app/tools/*.py 实现工具类，保持 run(arguments, step) -> (result, ToolEvent)。
4 tool_registry.py 把工具实例加入 self.tools。
5 tests/test_agent_loop.py 用 FakeModelClient 断言新工具会被调用，工具观察会进入第二轮模型输入。

检查点测试

先自己回答，再展开答案。你能说清这几个问题，就已经掌握 Agent Loop 主线了。

1. 为什么模型输出工具调用后，还要再调用一次模型？
因为工具只返回结构化观察，最终给用户的话术仍由模型生成。Loop 把 tool_call 和 tool observation 追加到 messages，下一轮模型基于观察生成 final_answer。
2. iot_state 和 tool_events 的区别是什么？
iot_state 是前端业务面板要消费的当前摄像头控制状态；tool_events 是调试轨迹，记录第几步调用了什么工具、输入输出和耗时。
3. 如果模型输出了未知工具名，会发生什么？
ToolRegistry 找不到工具，会返回 failed ToolEvent。当前实现为了满足 ToolName 枚举，把 event 的 tool_name 标成 final_answer，输出里会带 unknown tool 错误。
4. 为什么学习时建议先跑 tests/test_agent_loop.py？
因为这个测试用 FakeModelClient 和 FakeToolRegistry，完全不依赖 Ark CLI、网络搜索或 VikingDB。它隔离了 Agent Loop 本身，适合先理解控制流。
