一句话定位

S20

通用 Coding Agent Harness

S20 的核心目标是演示 Claude Code 类 agent 的完整外壳：工具、权限、hooks、任务、subagent、队友线程、 worktree、MCP、记忆、压缩、后台任务、cron 全部挂到一个循环上。

  • 入口：命令行交互
  • 核心：Anthropic native tool_use/tool_result 循环
  • 能力：通用代码执行与协作机制
  • 状态：本地文件、线程、消息总线
HomeGuard

领域受控的安防 App Agent

当前项目的核心目标是给安防 C 端 App 提供多轮对话、联网搜索、VikingDB 视频检索和 IoT 模拟控制。 它刻意收窄工具边界，把模型输出限定为结构化 JSON，再由后端校验和执行。

  • 入口：FastAPI /api/chat + React 页面
  • 核心：Ark CLI/SDK 适配层 + JSON 决策循环
  • 能力：业务工具，不开放 shell/file 写入
  • 状态：进程内会话 + 前端结构化展示
关键判断： S20 是“让模型操作开发环境”的 agent；当前项目是“让模型在安防业务边界内做路由与参数生成”的 agent。 因此 S20 的复杂度主要在 harness，当前项目的复杂度主要在业务工具、结构化响应和产品集成。

循环结构

S20 的循环：工具原生、动态、长循环

用户输入 UserPromptSubmit hook、cron/background 注入
准备上下文 memory、skills、MCP、压缩、transcript
LLM + tools Anthropic native tool schemas
执行 tool_use permission hook、handlers、MCP、background
tool_result 回填 继续循环，直到没有 tool_use

当前项目的循环：JSON 决策、业务受控、最多 3 步

API 请求 读取 conversation_id 和历史消息
模型生成 JSON final_answer 或 tool_call
ToolRegistry 执行业务工具 IoT、联网搜索、VikingDB 视频检索
结构化响应 assistant_message、iot_state、video_results、tool_events

差异矩阵

下面按实现维度展开。越靠前的差异越影响架构判断，越靠后的差异更多是工程成熟度和产品形态差异。

维度 S20 Comprehensive Agent 当前 HomeGuard Agent 影响
产品目标 教学型 coding agent，目标是把 Claude Code 类机制放进一个可运行 harness。 安防 C 端 App 的业务 Agent，目标是聊天、联网搜索、视频检索和模拟 IoT 控制。 方向不同 S20 追求通用能力，当前项目追求业务闭环和可控输出。
模型协议 直接使用 Anthropic messages.create(..., tools=...)，模型返回原生 tool_use block。 通过 Ark CLI/SDK 让模型输出 JSON 对象：final_answer 或 tool_call。 核心差异 S20 依赖模型供应商的工具协议；当前项目把工具协议做成业务 JSON 合约。
循环长度 while True 循环，直到模型不再返回 tool_use。还有 max_tokens、reactive compact 等恢复路径。 AgentLoop.max_steps = 3，三轮内必须得到最终回答，否则返回兜底文案。 风险控制 当前项目更适合 App 请求；S20 更适合长任务自动执行。
工具集合 内置 27 个工具：shell、文件、todo、task、cron、teammate、worktree、MCP 等。 3 个业务工具：web_search、video_search、iot_control。 边界不同 当前项目通过少量白名单工具降低安全和产品风险。
工具分发 assemble_tool_pool() 每轮动态合并内置工具和已连接 MCP 工具。 ToolRegistry 初始化时注册固定工具字典，运行期不动态扩展。 扩展方式 S20 适合插件化；当前项目适合稳定业务 API。
安全模型 通过 PreToolUse hook 做权限判断，拦截危险 bash、越界写文件、破坏性 MCP 工具。 不暴露 shell/文件工具；通过 Pydantic schema、ToolName 枚举和工具内校验限制输入。 安全重心 S20 是高权限工具前置审批；当前项目是能力白名单加参数校验。
Prompt 组装 每轮动态生成 system prompt，注入当前时间、skill catalog、memory、MCP、活跃 teammate。 静态 SYSTEM_PROMPT + TOOL_SPEC，每次请求固定拼接。 上下文能力 当前项目更稳定，但缺少动态能力描述和长期记忆注入。
记忆与状态 使用 .memory、.tasks、.mailboxes、.scheduled_tasks.json 等本地文件状态。 使用 InMemorySessionStore 保存 user/assistant 历史；工具观察只在一次 run() 内回填给模型。 持久化差异 当前项目重启会丢会话，且跨轮不会保留完整工具轨迹。
上下文压缩 包含 tool result budget、snip compact、micro compact、模型总结、prompt-too-long reactive compact。 暂无压缩管线；依赖短历史和最多 3 步控制上下文规模。 长期会话 如果未来支持长对话或多次视频检索，当前项目需要补上下文管理。
多 agent 有一次性 subagent 和持久 teammate 线程，通过 MessageBus、任务板和 plan approval 协作。 单 AgentLoop 同步处理单次请求，不做 subagent 分工。 复杂度取舍 当前业务场景暂时不需要 coding-agent 式协作。
错误恢复 处理 429、529、fallback model、max_tokens 升级、continuation、prompt too long 压缩重试。 Ark CLI/SDK 主要处理超时、认证过期、JSON 解析失败；ToolRegistry 捕获工具异常并返回 failed ToolEvent。 鲁棒性 当前项目在模型重试和 fallback 上还比较轻。
观测与 UI 以终端输出、hooks 日志和 transcript 文件为主。 后端返回 ToolEvent，前端 ToolTrace 折叠展示 input/output/status/耗时。 产品体验 当前项目更适合 C 端/调试页面；S20 更偏命令行工程师体验。

while True:
    inject_cron_and_background()
    prepare_context(messages)
    tools, handlers = assemble_tool_pool()
    response = call_llm(messages, tools)
    if not has_tool_use(response.content):
        trigger_hooks("Stop", messages)
        return
    for block in response.content:
        blocked = trigger_hooks("PreToolUse", block)
        output = blocked or call_tool_handler(handlers[block.name], block.input)
        trigger_hooks("PostToolUse", block, output)
        results.append(tool_result(block.id, output))
    messages.append({"role": "user", "content": results})

对当前项目的借鉴建议

不建议直接把 S20 全套机制搬进 HomeGuard。当前项目是面向 C 端业务的 Agent，安全边界应该继续保持窄。 更合理的做法是只吸收那些能提升稳定性和可观测性的机制。

优先引入：统一 hook / trace 管线

当前已有 ToolEvent，可以借鉴 S20 的 PreToolUse/PostToolUse/Stop 思路，把参数校验、耗时、错误、用户可见说明都统一成事件。这样后续新增工具不会散落日志逻辑。

优先引入：动态工具说明但保持白名单

S20 的 assemble_tool_pool() 很适合插件化。当前项目可以不开放动态 MCP，但可以把 TOOL_SPEC 从静态字符串改成由工具 schema 自动生成，避免新增工具时忘记同步 prompt。

中期引入：会话和工具轨迹持久化

现在 session_store 是进程内存，且跨请求只保存 user/assistant 终态。建议把完整工具观察存储到 SQLite、Redis 或业务数据库，方便长对话、问题复盘和效果评估。

中期引入：模型错误恢复与 fallback

可以借鉴 S20 的重试、fallback model、max_tokens 恢复，但要做成服务端策略，而不是 CLI 交互式策略。 对 C 端 App 来说，超时预算和用户反馈比无限自恢复更重要。

暂不建议：shell、文件写入、worktree、teammate

这些是 coding agent 的关键能力，但对安防 App 的线上 Agent 来说风险大于收益。除非未来做内部运维/研发助手， 否则不应出现在 C 端链路。

架构提醒： 如果未来要支持“视频检索后继续追问上一段视频”“跨轮追踪家庭设备状态”“长期用户偏好记忆”，当前的进程内会话和静态 prompt 会成为主要瓶颈。这时应优先补持久化、上下文摘要和可审计 trace，而不是先补多 agent。

结论

保留

当前项目的受控业务边界

三个业务工具 + JSON 合约 + Pydantic 校验，是适合 C 端安防 App 的实现方式。它牺牲了通用性，换来了可预测性。

补强

trace、schema、持久化

从 S20 借鉴 hook、动态 schema、错误恢复和上下文管理，会比引入 subagent/worktree 更有实际价值。

避免

不要复制 coding-agent 高权限工具

shell、文件写入、worktree、MCP destructive tool 适合开发者工具，不适合直接进入 C 端业务 Agent。
