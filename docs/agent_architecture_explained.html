给非技术同学看的版本

这个 Agent 到底怎么工作？

一句话：它不是固定关键词机器人，而是一个“模型先判断、后端再执行白名单工具、前端实时展示过程”的流式 Agent。

模型负责想 判断要不要调用工具，并生成严格 JSON 决策。
后端负责控 只执行注册过的工具，校验参数，记录结果。
前端负责看 边回答边展示状态、工具轨迹和摄像头状态。

Architecture

整体采用什么架构？

这是一个模型主导的工具型 Agent。模型不直接改摄像头状态，也不直接操作前端；它只能输出一个受约束的 JSON 决策。后端拿到决策后，再按白名单执行工具。

模型决策 + 后端工具执行 + SSE 实时返回
01 Frontend

用户界面

React 页面负责聊天、会话列表、摄像头状态面板和处理过程展示。

02 API

流式接口

FastAPI 只保留 /api/chat/stream，用 SSE 把过程一条条推给前端。

03 AgentLoop

编排大脑

最多跑 3 个决策步：问模型、执行工具、回填结果、生成答案。

04 Model

Ark SDK

火山方舟 SDK 负责模型调用：先出 JSON 决策，再流式出最终文字。

05 Tools

工具白名单

只有 web_search、video_search、iot_control 三种外部能力可执行。

06 Memory

会话上下文

后端保留进程内消息，前端也把会话存在 localStorage 里。

Runtime Flow

用户说一句话后，会发生什么？

你可以把它理解成一个流水线：用户说话，模型先选路，后端执行必要动作，结果再交给模型组织成自然语言。

最多 3 轮工具决策
1

接收用户目标

前端把用户消息、conversation_id 和客户端历史发到流式接口。

2

打包上下文

后端把系统提示词、工具说明、历史消息和当前消息放在一起。

3

模型做 JSON 决策

模型只能选择 final_answer 或 tool_call，不能自由发挥执行动作。

4

如果要工具

后端通过 ToolRegistry 找工具，执行后得到 tool_result 和业务状态。

5

Observation 回填

工具结果会变成 tool 消息，重新放回模型上下文，让模型基于事实继续。

6

生成最终回复

模型根据草稿、工具结果和上下文，流式输出用户能读懂的中文答案。

7

前端实时展示

answer_delta 拼成回答，tool_result 更新 IoT 面板和视频结果。

8

会话持久化

收到 final 后，后端保存用户和助手消息，前端保存当前会话状态。

Decision Protocol

为什么它不像普通聊天机器人？

普通聊天机器人直接生成一段话。这个 Agent 会先生成“可被程序理解的动作协议”，后端校验后才执行，所以更可控、更容易展示过程。

自由文本不驱动业务动作

路径 A：不需要工具，直接回答

比如用户问“你能做什么？”或闲聊，模型输出 final_answer。

{
  "type": "final_answer",
  "answer": "我可以帮你查视频、联网搜索，也可以模拟控制摄像头。",
  "iot_action": "none"
}

路径 B：需要工具，先调用工具

比如用户要“把摄像头转向门口”，模型输出 tool_call。

{
  "type": "tool_call",
  "tool_name": "iot_control",
  "arguments": {
    "device_id": "camera_living_room",
    "action": "move",
    "target": "front_door"
  },
  "reason": "用户要求摄像头转向门口"
}

Tools

它会执行哪些工具？

当前 ToolRegistry 注册了三个工具。模型可以请求它们，但真正执行权在后端；未知工具会被拒绝并记录失败事件。

工具是白名单，不是随便调
web_search

联网搜索

用于天气、新闻、政策、实时信息。天气优先走 wttr，普通搜索走 DuckDuckGo，必要时回退 Bing。

输入query, top_k
输出标题、摘要、URL
状态success / failed / skipped
video_search

视频检索

用于查监控片段。它调用 VikingDB 多模态检索，返回视频 ID、文本描述和分数。

输入query, limit
输出f_id, f_text, score
依赖VikingDB AK/SK
iot_control

IoT 模拟控制

用于移动摄像头、打开隐私遮蔽、恢复摄像头画面。注意：这里只是模拟，不接真实设备。

输入device_id, action, target
动作move / privacy_mask / none
输出IotState + ToolEvent

States

它大概有哪些状态？

这个项目里“状态”分三类：Agent 流程状态、IoT 业务状态、工具执行状态。前端能看到的处理过程主要来自这些状态。

状态让 Agent 可观测
01用户意图
02上下文打包
03模型决策层
04动作协议
05工具能力层
06Observation 回填
07最终回复
08前端可观测
IotState

摄像头业务状态

  • iot_action：none、move、privacy_mask
  • status：idle、simulated_success、validation_failed、tool_error
  • target：front_door、left、camera_on 等目标
ToolEvent

工具执行状态

  • success：工具执行成功
  • failed：参数错误、外部服务失败或未知工具
  • skipped：工具被配置关闭，比如 web_search disabled
SSE Event

前端接收的流事件

  • session、status、tool_call、tool_result
  • reasoning_delta、answer_delta、model_meta
  • final、error、done

What The User Sees

前端到底展示了什么？

前端不是只显示最后一句话。它会把后端流式事件拆开，分别更新消息、处理过程、IoT 面板、工具轨迹和视频结果。

聊天内容

answer_delta 会不断追加到助手消息里，用户看到的是“边生成边显示”。

处理过程

status、tool_call、tool_result、model_meta 会进入每条助手消息的 processEvents。

摄像头面板

tool_result 和 final 里的 iot_state 会驱动右侧模拟摄像头状态。

视频结果

video_results 来自 video_search，前端可以用它展示命中的片段描述。

错误提示

error 事件会显示到当前助手消息上，不会假装已经成功。

会话列表

前端用 localStorage 保存多个会话，后端收到 final 后保存本轮对话。

Safety & Limits

当前边界和保护机制

这个 Demo 已经有一些基本保护，但它仍然是演示原型，不是完整生产系统。

已有保护

  • 模型必须输出 JSON 决策，不能用自然语言直接驱动业务动作。
  • 工具由 ToolRegistry 白名单控制，未知工具会失败。
  • IoT 参数通过 Pydantic 校验，非法 action 会 validation_failed。
  • AgentLoop 最多 3 步，避免无限循环调用工具。
  • 最终回答提示词要求：工具失败要说明原因，视频结果不能编造。

当前限制

  • IoT 只是模拟，不会真的控制摄像头或设备。
  • 后端会话是进程内存储，重启会丢失。
  • VikingDB 视频搜索需要提前配置 AK/SK、collection 和 index。
  • 联网搜索依赖外部网络，可能超时或失败。
  • 生产级权限、审计、真实设备绑定还没有做。
最重要的一点： 这个 Agent 的核心价值不是“能聊天”，而是“能把模型决策、工具执行、状态变化、最终回复全部摊开给人看”。这让它更适合做安防 App 里的可解释智能助手 Demo。
