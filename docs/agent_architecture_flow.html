LLM Agent Architecture Map

不是普通 RAG，而是一个受控的安防业务 Agent

这个项目的核心升级，是把大模型从“回答生成器”提升为“任务决策中心”。模型负责理解用户意图、选择下一步动作、融合工具结果；后端负责约束动作空间、执行工具、校验参数和保存可观测事件。

一句话架构摘要

用户自然语言输入进入 Agent 后，模型先输出结构化决策。如果需要外部能力，后端执行白名单工具，并把 observation 回填给模型。最终回答必须基于真实工具结果，而不是凭空生成。

面试关键词

轻量 ReAct、工具调用、结构化输出、受控执行、Observation 回填、业务能力编排。

核心边界

模型只做决策，不直接操作设备；工具只在后端执行，IoT 当前是校验后的模拟控制。

整体架构：把“自然语言”转成“可控业务动作”

从大模型 Agent 的角度看，这个系统不是一个检索增强问答链路，而是一个“上下文管理 + 模型决策 + 工具执行 + 结果回填 + 最终表达”的闭环。每一层只承担一个清晰职责。

01 用户意图

用户只说目标，例如查视频、问天气、移动摄像头，不需要知道底层工具。

02 上下文打包

系统合并角色约束、工具规格、历史对话和当前消息，形成模型可判断的输入。

03 模型决策层

LLM 判断下一步是直接回答，还是调用 web、video、iot 等外部能力。

04 动作协议

模型必须输出 JSON 决策，不能用自由文本直接驱动业务动作。

05 工具能力层

后端按白名单执行联网搜索、视频语义检索、IoT 模拟控制。

06 Observation 回填

工具返回结果、状态和耗时，作为事实依据重新交给模型。

07 最终回复

模型基于工具 observation 生成自然语言，不编造未返回的视频或设备状态。

08 前端可观测

页面展示回答、工具轨迹、IoT 状态和视频结果，让 Agent 过程可解释。

Agent 决策闭环：Reasoning + Action + Observation

当前框架最重要的能力，是允许模型在受控空间里动态选择动作。它不是每次都固定检索，也不是每次都调用同一个工具，而是根据用户目标做路由和参数生成。

  1. 1
    理解意图 模型读取系统角色、工具说明、历史上下文和用户请求，判断任务类型。
  2. 2
    选择动作 模型输出结构化 JSON，动作只能是 final_answer 或白名单工具调用。
  3. 3
    执行工具 后端校验参数，执行工具，生成统一的 ToolEvent 和业务状态。
  4. 4
    回填观察 工具结果以 observation 形式回到模型上下文，下一轮模型基于事实继续判断。
  5. 5
    生成回复 最终表达和执行决策分离，回答必须引用已知工具结果和校验状态。

和普通 RAG 的区别：从“查资料回答”升级为“业务动作编排”

        

          普通 RAG 的核心是把相关资料检索出来，让模型基于资料回答。当前 Agent 框架保留了检索思想，但把检索变成了工具之一，并且引入了动作选择、状态管理和工具执行边界。

它为什么看起来更智能：智能来自受控闭环，不是来自更复杂的代码

        

          这个项目的智能感来自几个架构层面的组合：模型有决策空间，工具给它外部能力，协议让动作可控，observation 让回答基于事实，事件流让过程可观察。

01

            
模型作为决策中心

            
模型理解自然语言里的隐含目标，例如“昨晚门口有人吗”更像视频检索，“今天适不适合出门”更像实时搜索。

          

          

            
02

            
工具作为能力插件

            
工具把外部世界接进来。未来增加门锁、告警、家庭成员识别，只需要扩展工具能力层。

          

          

            
03

            
JSON 协议作为边界

            
模型不能随意生成“我已经控制设备”，它必须先输出动作，再由后端校验和执行。

          

          

            
04

            
Observation 作为事实回路

            
模型不是靠想象完成任务，而是等待工具返回，再基于返回结果继续生成答案。

          

          

            
05

            
安全边界作为产品底线

            
安防场景不能把模型自由输出直接变成设备动作，后端白名单和参数校验是必要边界。

          

          

            
06

            
事件流作为调试窗口

            
前端能看到 Agent 正在分析、调用什么工具、工具是否成功，让智能过程可解释。

          

        


        

          关键判断
          

            当前框架的升级点不是“用了一个更大的模型”，而是把模型放在一个有工具、有协议、有状态、有反馈的闭环里。智能感来自模型决策能力和后端执行能力的组合。
          

        

      


      

        
面试时可以这样收束

        

          下面这段可以直接作为口播结尾，用来把 Agent 架构、RAG 对比和业务价值合在一起。
        


        

          
interview.script

          
这个项目和普通 RAG 最大的区别，是它不只是“检索一段资料再回答”，而是把大模型放在了一个受控的业务执行闭环里。

普通 RAG 更像知识问答系统，核心能力是找到相关上下文。我的这个 Agent 更像业务编排层，用户说一句自然语言，系统要判断到底是直接回答、联网查实时信息、查视频语义库，还是模拟控制摄像头。

所以我做了一个轻量 ReAct 架构：模型负责理解意图和选择动作，后端负责工具白名单、参数校验和执行边界。每次工具执行后，结果都会作为 observation 回填给模型，最终回答必须基于这些真实结果生成。

这个设计让系统既有大模型的灵活理解能力，又不会让模型越过安全边界直接操作设备。对安防 C 端场景来说，这是一个比较关键的取舍：我们牺牲了一部分通用 Agent 的自由度，换来更强的可控性、可解释性和业务可落地性。
