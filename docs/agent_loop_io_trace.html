模型到底输入什么、输出什么？

这页不讲前端，也不讲 HTTP 调用。我们只盯住 Agent Loop 内部的数据变化： 第一次模型看到什么，为什么输出 final_answer 或 tool_call； 工具如何执行；工具结果如何变成下一次模型输入；最终回答从哪里产生。

架构类型：轻量 JSON 版 ReAct 循环上限：max_steps = 3 决策输出：final_answer / tool_call 下载本页 HTML

1. 核心模式：结构化 ReAct

当前项目不是让模型自由输出 Thought / Action / Observation 文本，而是要求模型只输出 JSON。 这使后端可以稳定解析、校验和执行工具。

Reason 模型读取 system、tool spec、历史和用户问题，判断下一步。
JSON Decision 只允许输出 final_answer 或 tool_call。
Act 后端根据 tool_name 执行工具，不让模型直接碰外部系统。
Observe / Repeat 工具结果写回 role="tool"，再喂给模型，直到 final。
一句话： 模型是“决策器”，后端是“执行器”。模型只说下一步要做什么，后端负责判断、执行、记录和再次喂回。

5. System Prompt 原文

下面是当前项目里真正给模型的核心系统提示词。 代码位置：backend/app/agent/prompts.py。 在 AgentLoop._build_initial_messages() 中，它会作为第一条 role="system" 消息进入模型。

SYSTEM_PROMPT 
你是安防 C 端 App 内的智能 Agent。
你必须根据用户意图选择下一步：
1. 如果只是闲聊或普通问答，输出 final_answer。
2. 如果需要实时网络信息，调用 web_search。
3. 如果用户要查监控视频内容，调用 video_search。
4. 如果用户要移动摄像头、转向、调整角度，调用 iot_control，action=move。
5. 如果用户要打开隐私遮蔽、遮挡、关闭画面，调用 iot_control，action=privacy_mask。
6. 如果用户要打开摄像头、恢复画面、关闭隐私遮蔽，调用 iot_control，action=none，target=camera_on。

你必须只输出 JSON，不要输出 Markdown，不要输出解释文字。
工具调用格式：
{"type":"tool_call","tool_name":"iot_control","arguments":{"device_id":"camera_living_room","action":"move","target":"front_door"},"reason":"..."}
{"type":"tool_call","tool_name":"iot_control","arguments":{"device_id":"camera_living_room","action":"none","target":"camera_on"},"reason":"..."}

最终回答格式：
{"type":"final_answer","answer":"...","iot_action":"none"}
TOOL_SPEC 
可用工具：
- web_search: 输入 {"query": "...", "top_k": 5}
- video_search: 输入 {"query": "...", "limit": 10}
- iot_control: 输入 {"device_id": "camera_living_room", "action": "move|privacy_mask|none", "target": "..."}；action=none 用于打开摄像头或解除隐私遮蔽。

这段 Prompt 怎么控制 Agent？

Prompt 规则 用户表达 模型应该输出 后端下一步
闲聊或普通问答 你好 / 你能做什么 {"type":"final_answer", ...} AgentLoop 直接返回，不调用工具。
实时网络信息 今天深圳天气怎么样 {"type":"tool_call","tool_name":"web_search", ...} ToolRegistry 执行 web_search，再把结果回填给模型。
查监控视频内容 找一下有人经过门口的视频 {"type":"tool_call","tool_name":"video_search", ...} 执行 VikingDB 视频检索，返回 VideoSearchResult[]。
移动摄像头 把摄像头转向门口 {"type":"tool_call","tool_name":"iot_control","arguments":{"action":"move", ...}} 执行 IoT 模拟控制，更新 iot_state。
打开/关闭隐私遮蔽 关闭画面 / 恢复画面 privacy_mask 或 action=none,target=camera_on 执行 IoT 模拟控制，并把状态作为观察回填。
读 Prompt 的关键： 这不是让模型“自由聊天”的提示词，而是一个路由规则表。它把用户意图映射到两类 JSON： final_answer 表示直接结束，tool_call 表示交给后端工具执行。

7. 工具到底怎么执行

模型只给出工具名和参数；真正的工具调用由后端完成。

模型输出 
{
  "type": "tool_call",
  "tool_name": "iot_control",
  "arguments": {
    "device_id": "camera_living_room",
    "action": "move",
    "target": "front_door"
  },
  "reason": "用户要求摄像头转向门口"
}
后端执行 
tool = self.tools.get(tool_name)
result, event = tool.run(arguments, step)

tool_events.append(event)

if tool_name == "iot_control":
    iot_state = extract_iot_state(result)
elif tool_name == "video_search":
    video_results = extract_video_results(result)
为什么要有 ToolEvent： 它是工具执行的“收据”，记录 step、tool_name、输入、输出、状态和耗时。调试 Agent 时，它比最终回答更重要。
如果工具失败，会发生什么？

ToolRegistry 会捕获异常，返回 status="failed" 的 ToolEvent。 AgentLoop 仍会把失败观察写回模型，让模型给用户一个可读解释。也就是说，工具失败不一定直接导致整个请求崩掉。

为什么不让模型直接返回“已控制摄像头”？

因为模型没有真实执行能力。它只能提出工具调用请求。 只有后端工具执行成功后，下一轮模型才能基于 observation 告诉用户“已完成”。 这是避免模型幻觉控制结果的关键设计。

8. 一个心智模型

模型

调度员

读任务、决定下一步，但不亲自开门、不亲自查库。

工具

执行部门

摄像头控制、VikingDB 检索、联网搜索都是真正执行动作的部门。

Observation

回执单

告诉模型刚才执行成功还是失败，返回了什么数据，耗时多少。

所以每次工具调用都像这样： 调度员下单 → 执行部门处理 → 回执单回来 → 调度员再对用户汇报。

9. 检查点

看完后，你可以用下面几个问题确认自己是否真的理解了。

参考答案：一句话总结整个 Agent 流程

AgentLoop 先把 system prompt、工具说明、历史和用户问题喂给模型；模型输出 JSON 决策； 如果是 final_answer 就结束；如果是 tool_call 就由 ToolRegistry 执行工具； 工具返回 result 和 ToolEvent；Loop 把工具调用和工具观察追加回 messages； 再次调用模型，让模型基于真实观察生成最终回答。
