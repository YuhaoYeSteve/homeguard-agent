Core Frame

开场先给一个算法总定位

这个项目 = LLM 决策器 + 工具白名单 + Observation 回填 + 流式最终回答。

作为面试者，我会先把它定义为一个轻量 ReAct 风格的工具增强 Agent。它不只是聊天，而是通过大模型判断下一步动作：直接回答，或调用 web_search、video_search、iot_control。工具执行结果再作为 observation 回填给模型，最终生成 grounded 的自然语言回复。

Q1
面试官问

这个项目更接近哪种 Agent 范式？

我会这样回答

这个项目更接近轻量级 ReAct / Tool-Use Agent。它不是单纯 ChatBot，也不是完整的 Plan-and-Execute 多步规划系统。它的核心是让 LLM 在每一轮里做结构化决策：要么输出 final_answer，要么输出 tool_call。后端执行工具后，把工具结果作为 observation 放回上下文，模型再继续决策或生成最终回答。

所以算法结构可以概括为：Thought-like decision、Action、Observation、Final Answer。只是这里没有直接暴露 chain-of-thought，而是把决策压缩成 JSON 协议。

ReAct Tool-Augmented LLM Structured Decision
考察点

候选人是否知道 Agent 不是“调一次模型”，而是模型、动作、观察、状态组成的循环。

追问可答

如果任务更复杂，比如“检查昨晚异常并恢复摄像头”，才更需要 Plan-and-Execute 或 planner。

Q2
面试官问

ReAct 的 Reasoning、Action、Observation 在项目中分别是什么？

我会这样回答

Reasoning 对应模型对用户意图和上下文的判断，但项目里没有把完整思维链直接暴露，而是让模型输出受约束的 JSON 决策。Action 对应 tool_call，比如调用 iot_control 或 video_search。Observation 对应工具执行结果，包括工具状态、输出、耗时、IoT 状态或视频结果。Final Answer 是模型基于这些 observation 再生成的自然语言回复。 

用户目标
  -> LLM JSON 决策
  -> tool_call
  -> tool_result / observation
  -> LLM final answer
考察点

是否能把抽象论文范式映射到实际系统，而不是只背 ReAct 名词。

追问可答

该项目更强调可控动作协议，而不是展示模型内部推理。

Tool Decision

工具选择是最核心的算法问题

这个项目的 Agent 能不能用，关键不是页面多漂亮，而是模型能否稳定判断：什么时候不用工具，什么时候查网络，什么时候查视频，什么时候模拟控制摄像头。

Q3
面试官问

这个 Agent 怎么选择工具？错误类型有哪些？

我会这样回答

目前主要是 LLM-based routing：系统提示词给出工具说明和触发条件，模型根据用户意图输出 tool_call 或 final_answer。比如实时信息走 web_search，监控内容查询走 video_search，摄像头移动或隐私遮蔽走 iot_control。

错误大概有四类：第一，该调用工具但直接回答；第二，调用了错误工具；第三，工具对但参数错，比如 target 不稳定；第四，模糊意图没有追问而强行执行。

提升方式包括：更清晰的 tool schema、few-shot 路由样例、专门的路由评测集、低置信度澄清策略，以及必要时引入一个轻量 intent classifier 做前置路由。

考察点

是否知道工具路由是 Agent 成败的关键算法模块。

项目映射

该项目评测里已有 route_accuracy、tool_accuracy、args_accuracy，这说明作者意识到了路由与参数需要单独评估。

Q4
面试官问

为什么要让模型输出 JSON，而不是直接输出文字？

我会这样回答

因为 Agent 的动作必须可解析、可校验、可审计。自然语言适合给人看，但不适合直接驱动工具。JSON 决策相当于把 LLM 的语义理解结果转换成机器可执行协议。

在这个项目里，模型只能输出两类结构：final_answer 或 tool_call。tool_call 里必须有 tool_name、arguments 和 reason。后端再用 schema 和工具白名单判断是否能执行。这样可以降低模型自由发挥导致的误动作风险。 

{
  "type": "tool_call",
  "tool_name": "iot_control",
  "arguments": {
    "action": "move",
    "target": "front_door"
  }
}
考察点

是否理解 structured output / function calling 是 Agent 落地的稳定性基础。

追问可答

真正生产中最好使用模型原生 function calling 或 JSON schema 约束，而不是只靠 prompt 说“请输出 JSON”。

Q5
面试官问

Observation 回填为什么重要？里面应该放什么？

我会这样回答

Observation 是 Agent 从外部世界拿到的事实反馈。没有 observation，模型只能凭参数和常识回答，容易幻觉。这个项目在工具执行后，把 tool_name、status、result、event 作为 tool 消息回填给模型，让最终回答基于真实工具结果。

我认为 observation 至少要包含：工具输入、工具输出、执行状态、错误信息、耗时，以及业务状态。例如 video_search 要返回视频片段和分数；iot_control 要返回模拟状态；web_search 要返回来源和摘要。

考察点

是否懂 grounding：最终回答必须被工具结果约束，而不是模型自己想象。

追问可答

Observation 太长时需要摘要、裁剪、证据排序，避免占满上下文。

RAG & Video Retrieval

视频检索不是“数据库查一下”

从算法角度，video_search 是一个面向监控内容的检索增强生成路径。它先检索视频片段，再让模型基于检索结果回答。

Q6
面试官问

video_search 本质上是什么算法？如何改进效果？

我会这样回答

它本质上是 RAG 的检索阶段，只是知识源不是普通文档，而是视频片段的文本描述或多模态向量索引。用户 query 进入 VikingDB 做多模态检索，返回候选片段，包括 f_id、f_text、score 等。模型最终只能基于这些候选片段回答。

如果效果不好，我会分层排查：视频切片粒度是否合理，片段描述是否准确，embedding 模型是否适配安防场景，召回 top_k 是否合理，是否需要 rerank，低分结果是否应该拒答。

多模态检索 RAG grounding 召回 + 重排
考察点

是否能把 RAG 套到视频场景，而不是只说“向量数据库查一下”。

追问可答

召回解决“找得到”，重排解决“排得准”。监控视频还要处理时间、对象、事件、地点等结构信息。

Q7
面试官问

多轮上下文在这个 Agent 里起什么作用？

我会这样回答

多轮上下文主要用于指代消解和状态延续。比如用户第一轮说“把摄像头转向门口”，第二轮说“再往左一点”，模型需要从历史中知道“再”指的是同一个摄像头控制任务，而不是普通聊天。

当前项目是短期会话记忆，直接把历史消息加入模型上下文。更成熟的做法是维护结构化状态，例如当前设备、上次动作、上次目标、隐私遮蔽状态等；长对话则需要摘要和状态抽取，不能无限塞历史。

考察点

是否知道 memory 不是简单保存聊天记录，而是服务于下一步决策。

追问可答

长期记忆要区分用户偏好、设备绑定、历史事件，还要有隐私授权。

Q8
面试官问

这个项目有没有真正的规划能力？什么时候需要 planner？

我会这样回答

当前项目不是完整 planner，它更像单步或少步 ReAct loop。每一轮模型只决定下一步：直接回答或调用一个工具。max_steps=3 允许它工具调用后再回答，但没有显式生成一个多步骤计划。

当用户目标包含多个子任务时，就需要 planner。例如“检查昨晚有没有异常，找出相关片段，并把摄像头恢复到门口”。这需要先拆成视频检索、异常摘要、IoT 恢复三个子步骤，并在每步后检查结果。

考察点

是否能区分轻量 ReAct 和 Plan-and-Execute。

追问可答

planner 会提升复杂任务能力，但也增加延迟、成本和错误传播风险。

Q9
面试官问

从算法角度，这个 Agent 怎么降低幻觉和错误动作？

我会这样回答

这个项目主要靠三层约束：第一，结构化 JSON 决策，避免自由文本直接驱动动作；第二，工具白名单和参数校验，模型请求不等于后端执行；第三，最终回答必须基于 observation，尤其视频搜索没有结果时不能编造监控内容。

但 prompt 不是完整安全方案。生产中还需要 policy model 或规则策略做高危动作拦截，例如删除录像、关闭报警、偷偷打开卧室摄像头等请求，应该拒绝或要求二次确认。

考察点

是否知道安全不能只靠提示词，工具调用必须有策略层。

追问可答

Prompt injection 会诱导模型泄露系统提示词或越权调用工具，必须在模型外做权限与策略校验。

Evaluation

Agent 不是只看最终回答对不对

一个工具型 Agent 的评估要拆开看：路径是否正确、工具是否正确、参数是否正确、回答是否基于证据、安全边界是否守住。

Q10
面试官问

你会怎么评估这个 Agent 的算法效果？

我会这样回答

我不会只看最终回答准确率。这个项目至少要分层评估：route_accuracy 看路径是否正确，tool_accuracy 看工具是否选对，args_accuracy 看参数是否正确，answer_groundedness 看回答是否基于工具结果，safety_pass_rate 看危险请求是否被拦截。

另外还要看 latency、工具失败率、澄清率、幻觉率和成本。因为 Agent 的错误可能发生在每一层：路由错、工具错、参数错、检索错、总结错。

考察点

是否能设计 Agent 评测，而不是只拿聊天答案做主观判断。

项目映射

项目里已经有非视频评测模块，包含 route、tool、args、safety、answer 等维度。

指标 看什么 为什么重要
route_accuracy 直接回答、工具调用、安全拒绝是否选对 决定 Agent 第一跳是否走偏
tool_accuracy web_search / video_search / iot_control 是否选对 工具错了，后续再好也难救
args_accuracy device_id、action、target、query 是否正确 工具参数是动作执行的核心
groundedness 回答是否基于 observation 和检索结果 防止模型编造视频内容或工具结果
safety_pass_rate 危险请求是否拒绝或澄清 安防场景下安全优先级最高
Q11
面试官问

这个项目目前最大的算法短板是什么？

我会这样回答

我认为短板有四个。第一，工具路由主要依赖 prompt，稳定性有限。第二，缺少显式置信度和澄清策略，模糊意图可能会被强行路由。第三，视频检索链路还没有完整的召回、重排、低分拒答策略。第四，安全策略主要靠提示词和工具校验，还没有独立 policy layer。

从算法上，我会优先补强工具选择评测和参数归一化，然后做 RAG grounding 评测，再加入高危动作策略模型或规则层。

考察点

是否能诚实识别 demo 和生产 Agent 的差距。

加分点

能把“短板”转成明确算法优化路线，而不是泛泛说“继续优化 prompt”。

Q12
面试官问

如果让你从算法角度升级一版，你会怎么做？

我会这样回答

我会分三层升级。第一层是决策层：引入更严格的 function calling schema、few-shot 路由样例、置信度字段和澄清策略。第二层是检索层：优化视频切片、embedding、top_k、rerank 和低置信拒答。第三层是安全与评估层：加入 policy check、危险动作分类、prompt injection 测试集和 groundedness 评估。

如果要支持复杂任务，我会再加 planner，把“检查异常并恢复摄像头”这类复合请求拆成可验证子任务，每个子任务执行后都要观察和校验。

考察点

是否能提出结构化优化方案，而不是只说“换更强模型”。

追问可答

更强模型有帮助，但工具 schema、数据质量、检索评估、安全策略往往更决定系统上限。

One-Minute Answer

最后可以这样做 1 分钟总结

这个项目是一个安防场景下的轻量 ReAct 风格工具型 Agent。它用 LLM 做结构化决策，判断用户请求是普通回答、联网搜索、视频检索还是 IoT 模拟控制。模型输出 JSON action，后端通过工具白名单执行并校验参数，工具结果作为 observation 回填给模型，最终模型基于 observation 生成自然语言回答。

从算法角度看，核心挑战不是前端后端，而是工具路由准确率、参数生成稳定性、RAG 检索 grounding、多轮上下文理解、幻觉控制、安全策略和评估体系。如果升级生产版，我会优先增强 function calling schema、澄清策略、视频召回重排、policy check 和分层评测指标。
